FluidKit v0.1

fluid-engine-dev の知見をもとに作った、流体エフェクト生成 & 多ツール連携パイプライン。
SPH シミュレーション → Neural Fluid 学習 → Three.js ブラウザ表示 をワンストップで。

Python 3.12 PyTorch 2.6 + CUDA Three.js r158 fluid-engine-dev 1.3.3 RTX 3070

概要

FluidKit は以下の 4 層で構成されます。

① 物理シミュレーション層

fluid-engine-dev (C++) の SPH/FLIP/Level Set エンジンでリアルな流体データを生成。
pyjet なし環境向けに Python SimpleSPH も内蔵。

② データ変換層

XYZ / POS 形式のシミュレーション出力を Three.js が読める frames.json に変換。 粒子数・FPS・境界ボックスのメタデータも付与。

③ Neural Fluid 層

Graph Neural Network (Interaction Network) で「現在フレーム → 次フレーム」を学習。 訓練済みモデルの推論結果も Viewer でリアルタイム表示。

④ レンダリング層

Three.js + カスタム GLSL シェーダーで球形パーティクルをリアルタイム描画。 パーティクル / メッシュ切り替え、カラーマップ 4 種対応。

SPH 流体シミュレーション

SPH (Smoothed Particle Hydrodynamics) は流体を有限個の粒子で近似する手法。 連続体力学の偏微分方程式を粒子ごとの相互作用に変換して数値的に解く。

カーネル関数 (Poly6)

W(r, h) = (1 − (r/h)²)³    (r < h の場合)
W(r, h) = 0           (r ≥ h の場合)

Poly6 カーネルは 密度計算 に使用。距離に対して滑らかに減衰し数値安定性が高い。

密度推定

ρi = Σj W(|ri − rj|, h)

圧力（Tait EOS 線形近似）

pi = k (ρi − ρ0)
k: 剛性係数, ρ₀: 静止密度

運動方程式（Navier-Stokes の粒子版）

ai = g + apressurei + aviscosityi

apressurei = −Σj (pi+pj) / (2ρj) · W(rij) · r̂ij
aviscosityi = μ Σj W(rij) / ρj · (vj − vi)

時間積分（Symplectic Euler）

vel[i] += acc[i] * dt   # 速度を先に更新
pos[i] += vel[i] * dt   # 更新後の速度で位置を移動
前向きオイラーよりエネルギー保存性が高く、同じ計算コストで安定性が向上します。

fluid-engine-dev コアエンジン

C++ ライブラリ sph_sim.exe が内部で使うアルゴリズム群。

PCISPH（予測補正 SPH）

標準 SPH の「小さい Δt が必要」という問題を解決する反復スキーム。

  • 1
    外力・粘性のみで仮位置・仮速度を予測
  • 2
    密度誤差 ρ̃ᵢ − ρ₀ を計算
  • 3
    圧力を反復修正して密度誤差を閾値以下に収束
  • 4
    修正圧力で最終的な加速度を計算
標準 SPH の 5〜10 倍大きい Δt で安定動作。計算量が大幅に削減されます。

FLIP（Fluid-Implicit Particle）

粒子と格子のハイブリッド手法。粒子の詳細な挙動と格子の圧力ソルバーを組み合わせる。

圧力ソルバー（MGPCG）

多重格子前処理付き共役勾配法。非圧縮条件（∇·u = 0）を満足する圧力場を求める。

ポアソン方程式: ∇²p = (ρ/Δt) ∇·u

V サイクル（MG 前処理）: 細かい格子 → 制限 → 粗い格子 → 平滑化 → 延長 → 細かい格子
計算量: O(N log N)

Level Set 法

符号付き距離場 φ(x) で液体界面を暗示的に表現。

φ < 0 : 液体内部 | φ = 0 : 液体表面 | φ > 0 : 気相

移流: ∂φ/∂t + u·∇φ = 0 再初期化（FMM）: |∇φ| = 1 を維持

KNN グラフ構築

# 各粒子の k 近傍を動的に計算
diff = pos.unsqueeze(2) - pos.unsqueeze(1)   # (B,N,N,3)
dist = (diff**2).sum(-1)                    # (B,N,N)
_, idx = dist.topk(k, largest=False)          # (B,N,k)

メッセージパッシング

エッジ特徴: eij = [hi ‖ hj ‖ (rj−ri) ‖ ‖rj−ri‖]
メッセージ: mij = MLPedge(eij)
集約: aggi = Σj∈N(i) mij
更新: h'i = MLPnode([hi ‖ aggi])

損失関数

L = wpos · MSE(pred_pos, true_pos) + wsmooth · mean((pred[1:] − pred[:-1])²)
wpos=1.0, wsmooth=0.01

活性化関数（SiLU）

SiLU(x) = x · σ(x) = x / (1 + e−x) // ReLU より滑らか。物理量の連続変化に適している

球形パーティクルシェーダー

GPU の 1 ポイント = 1 スプライトを球形に見せる手法。ジオメトリ生成が不要で高速。

// 頂点シェーダー: 遠近感のある点サイズ計算
gl_PointSize = uSize * (300.0 / -mv.z);

// フラグメントシェーダー: 円形クリップ + ソフトエッジ
vec2 uv = gl_PointCoord - 0.5;
float d = length(uv);
if (d > 0.5) discard;
float alpha = 1.0 - smoothstep(0.3, 0.5, d);

Stam 安定流体（格子法）— stable_fluids.py

Jos Stam (1999) の安定流体ソルバーを numpy で完全ベクトル化。

メソッド旧実装新実装効果
_advect()O(N²) Python ループscipy.ndimage.map_coordinates60〜80x 高速化
_project()O(N² × 20) Python ループnumpy スライシング (赤黒 SOR)~15x 高速化
_diffuse()Python ループnumpy スライシング~12x 高速化
from FluidKit.solver import StableFluids2D

fluid = StableFluids2D(N=64, dt=0.05, visc=0.001)
fluid.add_velocity(32, 32, 1.0, 0.5)  # 外力を加える
fluid.step()                               # 1ステップ: 拡散→投影→移流
density_field = fluid.dens                 # (N+2, N+2) numpy 配列
scipy.ndimage.map_coordinates は C 実装の双線形補間で、 Python ループの逆追跡をノーコピーで置き換えます。 これが最大のボトルネックだったため、60〜80x の改善になります。

SPH + 空間ハッシュ近傍探索 — sph_solver.py

全粒子ペアの O(N²) 探索を SpatialHash で O(N) に削減。

近傍探索の計算量: 旧 (全ペア): O(N²) N=500 → 250,000 ペア 新 (空間Hash): O(N) N=500 → 平均 ~9 セル × 近傍数だけ 
# 空間ハッシュの仕組み
# セルサイズ = カーネル半径 h → 近傍は必ず 3×3=9 セル内に収まる
hash.build(positions)         # O(N) で全粒子をグリッドに登録
nbrs = hash.query(pos_i, h)  # O(k) で近傍だけ取得
カーネル用途特徴
Poly6密度計算距離の二乗で評価 (sqrt 省略で高速)
Spiky 勾配圧力力粒子が近づくと急峻な反発を生む
Viscosity ラプラシアン粘性力速度差をなめらかにする

サンドボックス実行 — runner.py / reproducibility.py

Judge0 パターンをシミュレーション実行に適用。

タイムアウト（クロスプラットフォーム）

Unix: signal.SIGALRM
Windows: threading.Timer で _TimeoutError を非同期 raise

再現性保証

各ステップの状態を SHA-256 でハッシュ化して記録。
同じ設定で再実行 → 同じハッシュ列 → 完全に同一の実行。 

from FluidKit.solver import run_simulation_sandboxed, SimulationRecord

result = run_simulation_sandboxed(
    config={"N": 64, "dt": 0.05, "n_frames": 100, "seed": 42},
    solver_factory=lambda cfg: StableFluids2D(**cfg),
    timeout_sec=300,
    audit_path="audit.json",  # SHA-256 監査ログを保存
)
# → {"status": "completed", "frames": 100, "state_hash": "...", "elapsed_ms": 42.3}
validate_config() が入力を事前検証し、範囲外の値（N > 512, dt > 1.0 など）を 安全なデフォルト値に置換します。悪意ある設定（dt=0.001, n_frames=10000 など）による DoS を防止。

次のフェーズ

Blender 連携

Neural Fluid 精度向上

Unity / Unreal 出力

VFX Graph / Niagara 向けにパーティクルデータを JSON / CSV でエクスポートするツールを追加。

既知の制限： Neural Fluid の自律ロールアウト（推論を連鎖）は誤差が蓄積するため、 長時間シミュレーションでは挙動が実機から乖離します。 短期予測（20〜30 フレーム）での使用を推奨します。
