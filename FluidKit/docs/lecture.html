システム全体像

FluidKit は「流体エフェクトを手軽に作成し、複数のツールに簡単に連携・挿入できる」ことを目標に設計した、 6 フェーズ構成のプロダクションパイプラインです。 物理シミュレーション・リアルタイムレンダリング・機械学習・ネイティブコード・3D ツール連携を すべてゼロから作り上げています。

⚗️ Phase 1 — Python SPH

純 Python の軽量流体シミュレーター。外部ライブラリゼロで Poly6 カーネル + Symplectic Euler を実装。3 プリセット対応。

🎨 Phase 2 — Three.js Viewer

ブラウザで動く 3D パーティクル Viewer。カスタム GLSL Point Sprite シェーダー + Additive Blending。4 カラーモード。

🔬 Phase 3 — C++ SPH

fluid-engine-dev の PCISPH ソルバー。8000+ 粒子で研究品質のシミュレーション。XYZ 形式の自動検出付き変換ツール。

🎬 Phase 4 — Blender

MCP 経由で Blender を自動操作。SPH データをアニメーションとして読み込み、HDRI 照明 + マテリアルで映像制作品質にレンダリング。

⚡ Phase 5 — WebAssembly

C++ SPH を Emscripten で 3.3KB の WASM にコンパイル。ブラウザ上でリアルタイム物理演算 + パラメータ調整。

🧠 Phase 6 — Neural Fluid

Graph Neural Network（Interaction Network）で流体挙動を学習。100 シム × 9900 サンプルで訓練、GNN が物理演算を近似。

アーキテクチャ設計

FluidKit のアーキテクチャは「データフォーマットによる疎結合」を中心に設計されています。 各フェーズは独立しており、frames.json という共通フォーマットを通じて連携します。

設計原則

疎結合（Decoupling）

各フェーズは JSON/NPZ ファイルを介して通信。Python が停止していても Viewer は動く。

漸進的強化（Progressive Enhancement）

Phase 1 だけでも動く。Phase 6 を追加すれば AI 化できる。各層を独立して差し替え可能。

再現可能性（Reproducibility）

シード値・パラメータをすべて記録。同じ入力から同じ出力を保証。

スケーラビリティ

粒子数・フレーム数・シミュレーション数をすべてパラメータ化。小規模で動いたら数を増やすだけ。

📐 フォーマット設計の理由
フラット配列 [x0,y0,z0, x1,y1,z1, ...] を選んだのは、Three.js の BufferGeometry に直接渡せるからです。 オブジェクト配列 [{x,y,z}, ...] にするとパース時間が 3〜5 倍に増加します。 大容量データは JSON より NPY/NPZ の方が効率的ですが、ブラウザ互換性のため JSON を採用しています。

Phase 1SPH 理論の基礎

SPH（Smoothed Particle Hydrodynamics）は1977年に天体物理学のために開発された 粒子ベースの数値流体力学手法です。メッシュを使わず、流体を有限個の「粒子」として表現します。

なぜ粒子ベースか？

手法特徴向いている用途
格子法（Eulerian）固定グリッド上で流体を計算風洞実験・気象シミュレーション
SPH（Lagrangian）流体とともに粒子が移動水しぶき・破砕・自由表面流れ
FLIP/PIC格子と粒子のハイブリッドゲームエンジンの流体（Houdini等）

SPH の基本アイデア

連続した流体を N 個の粒子で近似します。ある粒子 $i$ の物理量 $A$ は、 周囲の粒子 $j$ の値を「スムージングカーネル」で重み付けした和として計算します：

$$A(\mathbf{r}_i) = \sum_j \frac{m_j}{\rho_j} A_j \, W(|\mathbf{r}_i - \mathbf{r}_j|, h)$$
SPH 補間式。$W$ はカーネル関数、$h$ はカーネル半径（影響範囲）

カーネル関数（Poly6）

FluidKit では最もポピュラーな Poly6 カーネルを使用しています。 距離 $r$ が $h$（カーネル半径）以内の粒子だけが影響し、距離が離れるほど滑らかにゼロに近づきます：

$$W(r, h) = \begin{cases} \left(1 - \frac{r^2}{h^2}\right)^3 & (r < h) \\ 0 & (r \geq h) \end{cases}$$
Poly6 カーネル。$r$ は粒子間距離、$h$ はカーネル半径
Pythongen_sample.py — Poly6 カーネル
def _kernel(r, h):
    """Poly6 kernel (smoothed weight).

    r: 粒子間距離
    h: カーネル半径（影響範囲）
    戻り値: 重み（0〜1）
    """
    if r >= h:
        return 0.0           # 影響範囲外はゼロ
    q = 1.0 - (r / h) ** 2  # 距離が離れるほど小さくなる
    return q * q * q         # 3乗でゼロに近づく（C² 連続）
⚠️ C² 連続性が重要な理由
カーネルは 2 階微分（圧力項・粘性項の計算）に使われます。 カーネルが $C^2$（2回連続微分可能）でないと、力の計算で数値的不安定が起きます。 Poly6 は $r=h$ で $W=0$, $W'=0$, $W''=0$ となるため安定です。

シミュレーションの 1 ステップ

各タイムステップで以下の順に計算します：

  1. 密度計算 粒子 $i$ の密度を周囲粒子からカーネルで集計します。
    $$\rho_i = \sum_j W(|\mathbf{r}_i - \mathbf{r}_j|, h)$$
  2. 圧力計算（Tait EOS） 密度と「静止密度」の差から圧力を求めます。$k$ は剛性係数（stiffness）：
    $$p_i = k(\rho_i - \rho_0)$$
    密度 > 静止密度 → 圧力 > 0 → 粒子を押し広げる。密度 < 静止密度 → 引き戻す。
  3. 加速度計算（Navier-Stokes） 重力 + 圧力勾配力 + 粘性力 を足し合わせます：
    $$\mathbf{a}_i = \mathbf{g} + \sum_j \left[ -\frac{p_i + p_j}{2\rho_j} \frac{W}{r} \mathbf{r}_{ij} + \frac{\mu}{\rho_j} W (\mathbf{v}_j - \mathbf{v}_i) \right]$$
  4. 積分（Symplectic Euler） 速度を先に更新し、その速度で位置を更新します（エネルギー保存性が高い）：
    $$\mathbf{v}_i^{t+1} = \mathbf{v}_i^t + \Delta t \cdot \mathbf{a}_i$$ $$\mathbf{r}_i^{t+1} = \mathbf{r}_i^t + \Delta t \cdot \mathbf{v}_i^{t+1}$$
  5. 境界処理（AABB 反射） 境界を越えた粒子を押し戻し、速度を反転 × 反発係数で減衰させます。

コード全解説 — SimpleSPH クラス

Pythongen_sample.py — SimpleSPH.step()
def step(self, dt: float = 0.016):
    h, k, rho0, mu = self.h, self.k, self.rho0, self.mu
    n = self.n
    pos, vel = self.pos, self.vel

    # ───── Step 1: 密度・圧力 ─────
    densities = [self._density(i) for i in range(n)]
    pressures = [k * (d - rho0) for d in densities]
    #            └───────────────┘ Tait EOS
    #            密度が静止密度より高ければ正の圧力（押し広げる力）

    # ───── Step 2: 加速度 ─────
    acc = [[gx, gy, gz] for gx, gy, gz in [self.g] * n]
    #      ↑ まず重力で初期化

    for i in range(n):
        pi = pos[i]
        for j in range(n):        # O(N²) — 小規模では問題なし
            if i == j: continue
            pj = pos[j]
            dx = pi[0]-pj[0]; dy = pi[1]-pj[1]; dz = pi[2]-pj[2]
            r = math.sqrt(dx*dx + dy*dy + dz*dz) + 1e-8  # ゼロ除算防止
            if r >= h: continue   # カーネル半径外はスキップ（スパース性を活用）

            w = _kernel(r, h)

            # 圧力勾配力（両粒子の平均圧力を使う対称形式）
            pf = -(pressures[i] + pressures[j]) / (2.0 * densities[j]) * w / r
            acc[i][0] += pf * dx   # x成分
            acc[i][1] += pf * dy   # y成分
            acc[i][2] += pf * dz   # z成分

            # 粘性力（速度差を滑らかにする拡散項）
            dvx = vel[j][0]-vel[i][0]
            dvy = vel[j][1]-vel[i][1]
            dvz = vel[j][2]-vel[i][2]
            vf = mu * w / densities[j]
            acc[i][0] += vf * dvx
            acc[i][1] += vf * dvy
            acc[i][2] += vf * dvz

    # ───── Step 3: 積分 (Symplectic Euler) ─────
    xlo, xhi, ylo, yhi, zlo, zhi = self.bounds
    restitution = 0.3  # 反発係数（0.3 = 70%エネルギー損失）

    for i in range(n):
        # 速度を先に更新
        vel[i][0] += acc[i][0] * dt
        vel[i][1] += acc[i][1] * dt
        vel[i][2] += acc[i][2] * dt
        # 更新された速度で位置を更新
        pos[i][0] += vel[i][0] * dt
        pos[i][1] += vel[i][1] * dt
        pos[i][2] += vel[i][2] * dt
        # 境界反射
        if pos[i][0] < xlo: pos[i][0] = xlo; vel[i][0] *= -restitution
        if pos[i][0] > xhi: pos[i][0] = xhi; vel[i][0] *= -restitution
        # ... y, z も同様
⚠️ O(N²) 問題
全粒子ペアをループするため計算量は $O(N^2)$ です。 N=400 で 160,000 ペア、N=1500 で 2,250,000 ペア。 本番では空間ハッシュ（格子法）で $O(N)$ に高速化します（fluid-engine-dev が実装）。 学習用サンプルでは粒子数を 400 に抑えてこの問題を回避しています。

GLSL シェーダー — Point Sprite

通常の THREE.Points は正方形のピクセルを描画します。 Point Sprite はシェーダーで各パーティクルを球形に見せるテクニックです：

GLSLVertex Shader — パーティクルサイズと色計算
// Vertex Shader
attribute float speed;      // 各粒子の速度スカラー

void main() {
    vec4 mvPos = modelViewMatrix * vec4(position, 1.0);

    // 透視投影でのピクセルサイズ計算
    // カメラから遠いほど小さく見える
    gl_PointSize = uSize * (300.0 / -mvPos.z);

    gl_Position = projectionMatrix * mvPos;

    // Fragment Shader に渡す変数
    vSpeed = speed;
    vY     = position.y;    // 深度カラー用のY座標
}
GLSLFragment Shader — 球形 + カラーモード
// Fragment Shader
void main() {
    // gl_PointCoord: Point の中心を(0.5,0.5)とする UV座標
    vec2 uv = gl_PointCoord - 0.5;
    float dist = length(uv);

    // 距離 > 0.5 なら円の外 → 描画しない（正方形を球形に）
    if (dist > 0.5) discard;

    // ガウシアン風の輝度（中心が明るく、端が暗い）
    float alpha = exp(-dist * dist * 8.0) * uOpacity;

    // カラーモードに応じて色を決定
    vec3 col;
    if (uColorMode == 0) {
        // depth モード: Y座標 → 青〜白
        float t = clamp(vY / uBoundsMax.y, 0.0, 1.0);
        col = mix(vec3(0.1, 0.3, 1.0), vec3(0.9, 0.95, 1.0), t);
    } else if (uColorMode == 1) {
        // speed モード: 速さ → 青〜赤
        float t = clamp(vSpeed / 5.0, 0.0, 1.0);
        col = mix(vec3(0.0, 0.3, 1.0), vec3(1.0, 0.1, 0.0), t);
    }
    // ...

    gl_FragColor = vec4(col, alpha);
}

PCISPH アルゴリズム

PCISPH（Predictive-Corrective Incompressible SPH）は通常の SPH の最大の弱点である 「圧力の不正確さ」を解消します。

  1. 予測ステップ
    現在の力（重力・粘性）だけで次フレームの位置・速度を仮計算します。
  2. 密度誤差計算
    仮計算した位置で密度を評価し、目標密度との誤差を計算します。
  3. 圧力修正ループ
    誤差が収束するまで（通常 3〜5 回）圧力を修正します。この反復が「非圧縮性の保証」になります。
  4. 最終積分
    修正された圧力を使って最終的な位置・速度を更新します。

XYZ フォーマット自動検出

fluid-engine-dev は標準 XYZ と異なる「raw XYZ」を出力します。 xyz_to_json.py は両方を自動判別します：

Pythonxyz_to_json.py — フォーマット自動検出
def parse_xyz(path: Path):
    """
    2種類のXYZフォーマットを自動検出してパース。

    ┌─ 標準 XYZ ──────────────────────┐  ┌─ raw XYZ（fluid-engine-dev）──────┐
    │ 1500           ← 粒子数          │  │ 0.123 2.100 -0.045  ← 直接座標   │
    │ comment        ← コメント行      │  │ 0.456 1.987  0.123               │
    │ 0.123 2.100 -0.045              │  │ ...                               │
    │ ...                             │  └───────────────────────────────────┘
    └─────────────────────────────────┘
    """
    lines = path.read_text().strip().splitlines()

    try:
        n = int(lines[0].strip())      # 1行目が整数 → 標準フォーマット
        data_lines = lines[2: 2 + n]  # ヘッダー2行をスキップ
    except ValueError:
        data_lines = lines             # 整数でない → raw フォーマット（全行が座標）

    flat = []
    for line in data_lines:
        parts = line.split()
        if len(parts) >= 3:
            flat.extend([round(float(parts[0]), 5),
                         round(float(parts[1]), 5),
                         round(float(parts[2]), 5)])
    return flat

Phase 4Blender MCP 連携

Blender を「API として」使います。通常は手でクリックする操作を、 MCP（Model Context Protocol）経由でプログラムから自動実行します。

MCP とは

MCP は「外部プログラムが Blender の Python 環境に直接コードを送り込める」プロトコルです。 Blender のアドオンとして port 9876 で待ち受けし、受信した Python コードを exec() で実行します。

チャンク分割の理由

⚠️ MCP の制限
1回の MCP 呼び出しで送れるコードは有限であり、また実行時間にもタイムアウトがあります。 それ以上に重要なのが変数スコープ問題：別々の execute_blender_code 呼び出しは それぞれ独立したスコープで実行されるため、前の呼び出しで作ったローカル変数にアクセスできません。

変数永続化パターン — driver_namespace

Blender の bpy.app.driver_namespace はセッション全体で生存する辞書です。 チャンク間で変数を共有するためにこれを使います：

C++ API 設計 — SoA レイアウト

WASM では配列のメモリレイアウトが重要です。 SoA（Structure of Arrays）は CPU キャッシュ効率が高く、SIMD 最適化もかかりやすい：

C++sph_wasm.cpp — SoA レイアウト
// ❌ AoS (Array of Structures) — キャッシュ効率が悪い
struct Particle { float px, py, pz, vx, vy, vz; };
static Particle particles[MAX_N];  // メモリ: [px0,py0,pz0,vx0,...][px1,py1...]

// ✅ SoA (Structure of Arrays) — キャッシュフレンドリー
static float g_px[MAX_N], g_py[MAX_N], g_pz[MAX_N];  // 位置
static float g_vx[MAX_N], g_vy[MAX_N], g_vz[MAX_N];  // 速度
static float g_rho[MAX_N], g_prs[MAX_N];              // 密度・圧力
// メモリ: [px0,px1,px2,...][py0,py1,py2,...] — 連続アクセスが速い

// ──────────────────────────────────────────
//  公開 API（JS から呼ぶ関数）
// ──────────────────────────────────────────

EXPORT void sph_init(int n, int preset) {
    g_n = (n > MAX_N) ? MAX_N : n;   // 安全な上限チェック
    // プリセットに応じた初期配置
    switch (preset) {
        case 1:  initSmoke(g_n);  break;
        case 2:  initSplash(g_n); break;
        default: initDrop(g_n);   break;
    }
}

EXPORT float* sph_get_positions() {
    // SoA → インターリーブ形式に変換（Three.js が要求する形式）
    for (int i = 0; i < g_n; i++) {
        g_pos_buf[i*3+0] = g_px[i];
        g_pos_buf[i*3+1] = g_py[i];
        g_pos_buf[i*3+2] = g_pz[i];
    }
    return g_pos_buf;   // ポインタを JS に返す（ゼロコピー）
}
