① SPH 流体シミュレーション

SPH（Smoothed Particle Hydrodynamics） は流体連続体を有限個の粒子で近似し、 各粒子が持つ物理量（密度・圧力・速度）を近傍粒子の重み付き和として評価する手法。 偏微分方程式を格子なしで解けるため、自由表面・大変形流体に強い。

物理量 $A$ の位置 $\mathbf{r}$ における値：

$$A(\mathbf{r}) = \sum_j m_j \frac{A_j}{\rho_j} W(\mathbf{r} - \mathbf{r}_j,\, h)$$

Poly6 カーネル関数

カーネル関数 $W$ は粒子間の距離に応じた重みを決める。本実装では密度計算に Poly6 カーネル を使用：

$$W(r,\,h) = \begin{cases} \left(1 - \dfrac{r^2}{h^2}\right)^3 & r < h \\ 0 & r \geq h \end{cases}$$ 
def _kernel(r, h):
    if r >= h:
        return 0.0
    q = 1.0 - (r / h) ** 2
    return q * q * q   # (1 - (r/h)²)³
python
Poly6 は距離の 2 乗 を使うため、sqrt の計算を省略できます。数値安定性が高く密度計算に最適です。 圧力勾配にはより急峻な Spiky カーネル が適しますが、本実装では簡略化しています。

密度推定

粒子 $i$ の密度を、影響半径 $h$ 内の全近傍粒子の寄与で計算：

$$\rho_i = \sum_j W\!\left(|\mathbf{r}_i - \mathbf{r}_j|,\, h\right)$$

（質量を定数 1 に正規化したシンプルな形式）

圧力 — Tait EOS 線形近似

$$p_i = k\,(\rho_i - \rho_0)$$
記号意味実装変数
$k$剛性係数（大きいほど非圧縮に近い）stiffness
$\rho_0$静止密度（目標密度）rest_density

$\rho_i > \rho_0$ のとき $p_i > 0$（正圧：粒子を押し広げる）、 $\rho_i < \rho_0$ のとき $p_i < 0$（負圧：粒子を引き寄せる）。

Navier-Stokes 方程式の粒子離散化

各粒子の加速度を重力・圧力・粘性の 3 項に分解する：

$$\mathbf{a}_i = \underbrace{\mathbf{g}}_{\text{重力}} + \underbrace{\mathbf{a}_i^{\text{pressure}}}_{\text{圧力項}} + \underbrace{\mathbf{a}_i^{\text{viscosity}}}_{\text{粘性項}}$$

圧力勾配項（対称形式）

両粒子の圧力を平均することで作用・反作用が自動的に成立：

$$\mathbf{a}_i^{\text{pressure}} = -\sum_j \frac{p_i + p_j}{2\,\rho_j}\,W(r_{ij},h)\,\hat{\mathbf{r}}_{ij}$$

粘性項

速度差に比例する散逸力。流体の "ねばり" を表現：

$$\mathbf{a}_i^{\text{viscosity}} = \mu \sum_j \frac{W(r_{ij},h)}{\rho_j}\,(\mathbf{v}_j - \mathbf{v}_i)$$

時間積分 — Symplectic Euler

Symplectic Euler（半陰解法オイラー）は、速度を先に更新してからその速度で位置を更新する。 前向きオイラーよりエネルギー保存性が高く、同じ計算コストで安定性が向上する。

$$\mathbf{v}_i^{n+1} = \mathbf{v}_i^n + \mathbf{a}_i^n \,\Delta t$$ $$\mathbf{r}_i^{n+1} = \mathbf{r}_i^n + \mathbf{v}_i^{n+1} \Delta t \quad \leftarrow \text{更新後の速度を使う}$$ 
vel[i] += acc[i] * dt   # ① 速度を先に更新
pos[i] += vel[i] * dt   # ② 更新後の速度で位置を移動
python

境界処理（AABB 反射）

if pos[i][1] < y_min:
    pos[i][1] = y_min
    vel[i][1] *= -restitution   # 反発係数 0.3
python

実装ノート — SPHSolver 統合で発覚したバグ（2026-07）

上記の密度・圧力・粘性の式を、全ペア探索ではなく空間ハッシュ（セルサイズ = カーネル半径 $h$）で 近傍を $O(N)$ に絞り込んだ上で numpy ベクトル化した実装が solver/sph_solver.py の SPHSolver クラスである。ただしこのクラスはこれまでどのパイプラインからも 呼び出されたことがなく、ドキュメントに記載されていた「47倍高速化」という数値は、 一度も end-to-end で実行検証されていなかった。

tools/gen_sample.py の SimpleSPH（純 Python 二重ループ）を SPHSolver に置き換える作業の中で、近傍粒子が 2 個以上になるステップで必ず 例外が発生することが判明した。 

# _compute_forces() 内 — バグのあった行（修正前）
prs_force = -(
    mass[nbrs_arr] * prs_avg / rho[nbrs_arr, None]   # (k,) / (k,1) のつもりが…
    * self._W_spiky_grad(r_vecs, rs)                # grad: (k, dim)
).sum(axis=0)
python

mass[nbrs_arr] * prs_avg は shape (k,)。一方 rho[nbrs_arr, None] は rho[nbrs_arr]（shape (k,)）に None で軸を足したもので shape (k, 1) になる。ここで (k,) と (k, 1) を割り算すると、numpy のブロードキャスト規則により (k,) 側が暗黙に (1, k) として扱われ、結果は意図した (k, 1) ではなく (k, k) 行列になってしまう。この (k,k) 行列を shape (k, dim) の勾配ベクトルに掛け合わせようとして shape 不一致で ValueError となる（近傍数 k が 1 以下だと偶然 shape が壊れず素通りしていた）。 

# 修正後
prs_coeff = mass[nbrs_arr] * prs_avg / rho[nbrs_arr]   # (k,) — 先に係数として確定させる
prs_force = -(
    prs_coeff[:, None] * self._W_spiky_grad(r_vecs, rs)   # (k,1) * (k,dim) → (k,dim)
).sum(axis=0)
python
修正: 係数を先に (k,) の1次元配列として計算してから、 明示的に [:, None] で (k, 1) に reshape して勾配配列 (k, dim) にブロードキャストする形に変更（粘性力の計算も同様の構造だったため 同じパターンで修正）。独立した 2D/3D 再現テストと、修正前後で発散量・境界貫通の有無を 比較して検証した。

修正・統合後の実測（water_drop, N=1500, 10 フレーム）は 9.6 秒 → 1.72 秒（5.6 倍）。計画時点で期待した 100 倍には届かなかった — 空間ハッシュで近傍探索自体は $O(N)$ 化されているが、_compute_density / _compute_forces は「粒子ごとの Python ループ + 近傍数分の numpy 演算」という ハイブリッド構造であり、完全にはベクトル化されていないため。

実装ノート — 圧力投影のスパース CG 置換実験（不採用, 2026-07）

上記の MGPCG は多重格子前処理付き CG だが、本実装の stable_fluids.py / flip_solver.py の圧力投影は、固定 20 回反復の Gauss-Seidel／Jacobi ループで $\nabla^2 p = \frac{\rho}{\Delta t}\nabla \cdot \mathbf{u}$ を近似的に解いている。 より軽量な代替として、この固定反復ループを前処理なしの scipy.sparse.linalg.cg に置き換える実験を両ファイルで実施した。

境界条件（Neumann-copy 境界 / Dirichlet 境界）の導出は数値的に正しいことを検証済み （長時間 Gauss-Seidel 参照解と $10^{-12}$ オーダーで一致、投影後の発散量も旧実装と同等以上）。 しかし実際に使用しているグリッド規模（$N=64$〜$128$）では、疎行列の構築と CG ソルバー 呼び出しのオーバーヘッドが密配列スライス演算のコストを大きく上回り、 旧実装より最大 65 倍遅いという結果になった。

Level Set 法

液体表面を符号付き距離場 $\phi(\mathbf{x})$ で暗示的に表現する手法。 メッシュが不要で、トポロジーの変化（液滴の分裂・合体）を自然に扱える。

$$\phi(\mathbf{x}) \begin{cases} < 0 & \text{液体内部} \\ = 0 & \text{液体表面} \\ > 0 & \text{気相} \end{cases}$$

移流方程式

$$\frac{\partial \phi}{\partial t} + \mathbf{u} \cdot \nabla \phi = 0$$

再初期化（FMM）

移流後に $|\nabla \phi| = 1$（距離場の条件）が崩れるため、Fast Marching Method で符号付き距離場を再構築する。

FMM は界面からの伝播を一方向に行うため、反復法より高速 $O(N \log N)$ に再初期化できます。

セミラグランジュ移流

格子点 $\mathbf{x}$ の値を更新するために、その格子点から 過去にさかのぼって 流体粒子がどこから来たかを計算する（バックトレース）。

$$\mathbf{x}_{\text{back}} = \mathbf{x} - \mathbf{u}(\mathbf{x})\,\Delta t$$ $$\phi^{n+1}(\mathbf{x}) = \phi^n(\mathbf{x}_{\text{back}}) \quad \leftarrow \text{補間で取得}$$
CFL 条件（$|\mathbf{u}|\,\Delta t \leq \Delta x$）を超える $\Delta t$ でも 無条件安定。 補間精度（線形 or 三次スプライン）が精度を左右します。

KNN グラフ構築

各フレームで粒子位置から $k$ 近傍グラフを動的に構築する：

def knn_graph(pos, k=12):
    # (B, N, N, 3) — 全粒子ペアの差ベクトル
    diff = pos.unsqueeze(2) - pos.unsqueeze(1)
    # (B, N, N)   — ユークリッド距離の 2 乗
    dist = (diff ** 2).sum(-1)
    dist.diagonal(dim1=1, dim2=2).fill_(float('inf'))  # 自己除外
    _, idx = dist.topk(k, largest=False)  # (B, N, k)
    return idx
python
距離の 2 乗を使うため sqrt を省略。粒子位置が毎フレーム変化しても対応できる動的グラフ構築です。計算量は $O(N^2)$（粒子数 400 程度では十分高速）。
実装ノート（Phase 2, 2026-07）: 上記の knn_graph() は forward の たびに (B,N,N,3) の距離テンソルを生成するため、これを scipy.spatial.cKDTree による近傍インデックスの事前計算に置き換えた。学習データは collect_data.py が 事前計算し、推論は infer_v2.py がロールアウトの各フレームで計算する （推論時は次フレームの位置が未知のため事前計算できない）。model_v2.py の forward / predict_next は近傍インデックス nbr を 任意引数として受け取り、nbr=None の場合のみ旧 knn_graph() に フォールバックする形で後方互換を維持した。$N=400,\,k=16,\,B=8$ で forward 31.46ms→15.99ms（約2倍）、$N=4000$ で GPU メモリ 3450MB→2076MB（約1.66倍）削減。 旧 $O(N^2)$ 近傍集合との一致率 100%、モデル出力もビット一致を確認済み。

メッセージパッシング

各粒子の潜在ベクトル $\mathbf{h}_i$ を近傍粒子からのメッセージで更新するプロセス：

Step 1 — エッジ特徴の組み立て

$$\mathbf{e}_{ij} = \left[\mathbf{h}_i \;\|\; \mathbf{h}_j \;\|\; (\mathbf{r}_j - \mathbf{r}_i) \;\|\; \|\mathbf{r}_j - \mathbf{r}_i\|\right] \;\in\; \mathbb{R}^{2L+4}$$

2 粒子の潜在特徴 + 相対位置 + 距離スカラー。相対位置を含めることで回転不変性を近似的に実現。

Step 2 — メッセージ計算（EdgeMLP）

$$\mathbf{m}_{ij} = \mathrm{MLP}_{\text{edge}}(\mathbf{e}_{ij}) \;\in\; \mathbb{R}^{L}$$

Step 3 — 集約（Sum Aggregation）

$$\text{agg}_i = \sum_{j \in \mathcal{N}(i)} \mathbf{m}_{ij}$$

Step 4 — ノード更新（NodeMLP）

$$\mathbf{h}'_i = \mathrm{MLP}_{\text{node}}\!\left([\mathbf{h}_i \;\|\; \text{agg}_i]\right)$$

Step 1〜4 を $n_{\text{mp}} = 2$ 回繰り返す（遠くの粒子の情報を段階的に伝播）。

活性化関数 — SiLU

$$\mathrm{SiLU}(x) = x \cdot \sigma(x) = \frac{x}{1 + e^{-x}}$$
活性化関数特徴採用理由
ReLU$\max(0,x)$— 不採用：$x < 0$ で勾配消失
SiLU$x \sigma(x)$✓ 滑らか、負値も通す。物理量の連続変化に適合
GELU$x \Phi(x)$近似値だが SiLU とほぼ同等

損失関数

$$\mathcal{L} = w_{\text{pos}} \cdot \mathrm{MSE}(\hat{\mathbf{r}},\, \mathbf{r}^*) + w_{\text{smooth}} \cdot \frac{1}{N-1}\sum_{i=1}^{N-1}\|\hat{\mathbf{r}}_{i+1} - \hat{\mathbf{r}}_i\|^2$$
項重み役割
位置 MSE$w_{\text{pos}} = 1.0$次フレームの位置予測精度を直接最小化
平滑化正則化$w_{\text{smooth}} = 0.01$隣接粒子間の急激な位置変化を抑制。物理的に自然な運動を誘導

④ レンダリング層

Point Sprite — 球形パーティクルシェーダー

Three.js の THREE.Points は GPU の 1 描画コール = 1 ポイント。 各点を正方形スプライトとして描画し、フラグメントシェーダーで円形（球形）にクリップする。 ジオメトリシェーダー不要で数万粒子をリアルタイム処理できる。

頂点シェーダー — 遠近感のあるサイズ計算

uniform float uSize;
attribute float aSpeed;
varying float vSpeed;
varying float vY;

void main() {
    vSpeed = aSpeed;
    vY = position.y;
    vec4 mv = modelViewMatrix * vec4(position, 1.0);
    // カメラ距離 -mv.z に反比例してサイズを変化させる（遠近感）
    gl_PointSize = uSize * (300.0 / -mv.z);
    gl_Position  = projectionMatrix * mv;
}
glsl

フラグメントシェーダー — 球形クリップ & ソフトエッジ

varying float vSpeed;
varying float vY;

void main() {
    vec2 uv = gl_PointCoord - 0.5;   // 中心を (0,0) に
    float d = length(uv);
    if (d > 0.5) discard;           // 円形クリップ

    // smoothstep でエッジをぼかす → ソフトな球に見える
    float alpha = 1.0 - smoothstep(0.3, 0.5, d);

    vec3 col = colorDepth(vY);   // カラーマップを適用
    col += vec3(0.15) * (1.0 - d / 0.5);  // ハイライト
    gl_FragColor = vec4(col, alpha * uOpacity);
}
glsl

カラーマッピング（GLSL 内で完結）

全計算をシェーダー内で行うため、CPU-GPU データ転送がゼロ。

モード入力グラデーション用途
depthY 座標深青 → シアン → 白水深の可視化

時間ベースアニメーション制御

requestAnimationFrame の呼び出し間隔は可変（モニタリフレッシュレートやタブの背景化で変動）。 経過時間を 累積 して管理することでフレームスキップ・可変速再生に対応する。 

// 前フレームからの経過時間を累積
this._accTime += dt * this.playbackSpeed;

// シミュレーションの 1 フレーム分 (fpsDt) を消費するたびにフレーム更新
while (this._accTime >= fpsDt) {
    this._accTime -= fpsDt;
    this._updateFrame(nextFrame);   // GPU バッファを更新
}
javascript
単純に frameIndex = Math.floor(elapsed / fpsDt) とする方法と違い、 1 フレームの描画が重くても次のフレームをスキップせず正確に追跡できます。

⑤ アルゴリズム選定の理由

層採用アルゴリズム選定理由
Python シミュ SPH + Poly6 カーネル
Symplectic Euler		 実装がシンプル、pyjet なしで動作可能。Poly6 は sqrt 不要で高速
本格シミュ PCISPH / FLIP PCISPH は大 $\Delta t$ で安定。FLIP は粒子・格子の利点を両立
圧力ソルバー MGPCG $O(N \log N)$ 収束。大解像度格子でも実用的な速度
液体表面 Level Set + FMM 分裂・合体などトポロジー変化を自然に扱える
Neural Fluid モデル Interaction Network (GNN) 粒子ベースデータへの最も自然な帰納バイアス。グラフ構造が粒子間相互作用に対応
近傍探索 $O(N^2)$ 全対全（学習時） 粒子数 400 程度では GPU の並列演算で十分高速。実装が単純
最適化 Adam + ReduceLROnPlateau 適応学習率で安定収束。手動チューニングが少ない
レンダリング Point Sprite + GLSL シェーダー ジオメトリ不要。数万粒子でも 60fps 以上でリアルタイム描画
ブレンド Additive Blending Z ソート不要で高速。密度可視化に直感的
